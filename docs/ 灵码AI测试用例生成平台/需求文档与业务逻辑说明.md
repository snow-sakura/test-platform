## 一、项目概述

### 1.1 项目定位

灵码AI测试用例生成平台是一个基于 **AutoGen 多智能体框架** 的 AI 辅助测试用例生成系统。用户通过 Web 界面输入需求描述或上传需求文档（PDF/TXT），系统自动调度三个专业化 AI 智能体协同工作，生成结构化的测试用例，并经过自动评审流程输出高质量结果。

### 1.2 核心价值

- **提效**：将测试用例编写从小时级压缩到分钟级
- **提质**：多智能体协作 + 自动评审，确保覆盖率
- **易用**：Web 对话界面，支持文件上传，无需配置

### 1.3 目标用户

- 测试工程师：快速生成测试用例初稿
- 测试经理：评审和标准化测试用例
- 研发工程师：自测场景快速生成

## 二、功能需求

### 2.1 功能清单

| 编号 | 功能模块 | 功能描述 | 优先级 |
|------|---------|---------|--------|
| F-01 | 多智能体对话 | 用户输入需求，三个AI智能体协作生成测试用例 | P0 |
| F-02 | 文件上传解析 | 支持上传 PDF/TXT 文件并提取文本内容 | P1 |
| F-03 | SSE 实时流式响应 | AI 响应通过 SSE 实时推送到前端 | P0 |
| F-04 | 会话管理 | 创建管理多个独立对话会话 | P1 |
| F-05 | 消息时间轴展示 | AI 消息按时间线排列，不同智能体以不同颜色标识 | P1 |
| F-06 | 长内容折叠 | 超过 500 字符的 AI 响应自动支持折叠/展开 | P2 |
| F-07 | 自动重连 | SSE 连接断开后自动重连（最多 5 次） | P1 |
| F-08 | 连接状态提示 | 实时显示连接状态和 AI 处理状态 | P2 |
| F-09 | 暗黑模式适配 | 页面样式支持系统暗黑模式自动切换 | P3 |

### 2.2 功能详情

#### F-01：多智能体对话

**业务流程**：
1. 用户在前端输入需求描述（如"帮我编写登录功能的测试用例"）
2. 系统通过 HTTP POST 提交到后端 `/api/chat`
3. 后端触发 AutoGen 团队运行，三个智能体按 RoundRobin 顺序依次输出
4. 每个智能体的输出通过 SSE 推送到前端
5. 前端按智能体分组展示响应

**参与的智能体**：
| 智能体 | 角色 | 核心职责 |
|--------|------|---------|
| coordinator | 流程协调员 | 接收需求、分配任务、总结结果、控制流程 |
| test_writer | 测试用例编写员 | 根据需求编写详细测试用例（Markdown表格格式） |
| test_reviewer | 测试用例评审员 | 评审用例完整性、准确性、覆盖率，给出评分和建议 |

**终止条件**：
- `TextMentionTermination("TERMINATE")`：任一智能体输出包含 TERMINATE 时结束
- `MaxMessageTermination(30)`：最多 30 轮消息后强制结束

#### F-02：文件上传解析

**支持格式**：
| 文件类型 | 后端处理方式 | 前端处理方式 |
|---------|------------|------------|
| PDF | PyMuPDF 提取文本 | pdfjs-dist 提取文本 |
| TXT | UTF-8 解码 | file.text() 读取 |

**流程**：
1. 前端（`InputArea.vue`）：用户点击附件按钮 → 选择文件 → `parseFile()` 解析 → 将提取的文本暂存
2. 用户输入需求并发送 → 附件文本通过 `attachment` 字段随消息一起 POST
3. 后端（`sse_handler.py`）：将附件文本拼接到用户消息前（"附件内容：\n{附件}\n\n用户需求：\n{需求}"）
4. 智能体基于附件内容生成测试用例

#### F-03：SSE 实时流式响应

**通信协议**：

```
客户端 → 服务端 (POST /api/chat)
{
  "session_id": "uuid-string",
  "content": "用户输入的消息",
  "attachment": "文件提取的文本内容（可选）"
}

服务端 → 客户端 (SSE Events)
event: connected       → 连接建立
event: processing      → AI 开始处理
event: UserInputRequest → AI 请求用户额外输入
event: AgentResponse   → 智能体响应（含 source 标识）
event: done           → 对话完成
event: error          → 错误信息
```

#### F-04：会话管理

- 每次前端连接时通过 `POST /api/session` 创建新的会话
- 返回 UUID 格式的 `session_id`
- 后端内存中维护 `sessions` 字典，存储队列、团队实例、状态
- SSE 流断开时清理事件队列，但保留 session 和团队状态

#### F-05：消息时间轴展示

- AI 消息左侧显示垂直时间轴线和彩色圆点
- 不同智能体使用不同颜色：
  - 编写员（test_writer）：绿色 `#52c41a`
  - 评审员（test_reviewer）：橙色 `#fa8c16`
  - 协调员（coordinator）：紫色 `#722ed1`
  - 通用助手（assistant）：蓝色 `#1677ff`
- 用户消息右侧对齐，无时间轴

#### F-06：长内容折叠

- AI 响应字符数 > 500 时，消息气泡底部显示"展开全部"/"收起"按钮
- 折叠状态默认展开（`isCollapsed = false`），用户可手动切换
- 展开时显示全部内容，收起时限制高度为 150px 并带透明度渐变

## 三、业务逻辑详解

### 3.1 核心业务流程

```
用户输入需求 → POST /api/chat → 创建 TextMessage → team.run_stream()
                                                          │
                    ┌─────────────────────────────────────┼─────────────────────────────┐
                    │                                     │                             │
              coordinator                           test_writer                   test_reviewer
            (接收需求/分配任务)                    (编写测试用例)                  (评审测试用例)
                    │                                     │                             │
                    └─────────────────────────────────────┼─────────────────────────────┘
                                                          │
                                              收集所有智能体消息
                                                      │
                                              按 source 分组合并
                                                      │
                                          SSE 推送 AgentResponse (逐个智能体)
                                                      │
                                               SSE 推送 done 事件
```

### 3.2 智能体协作协议

**第1步 - 流程协调员（coordinator）**：
- 接收用户原始需求
- 理解需求范围，确认输入信息是否充分
- 将任务分配给测试用例编写员

**第2步 - 测试用例编写员（test_writer）**：
- 根据需求描述编写详细的测试用例
- 输出格式：Markdown 表格（用例编号、用例标题、前置条件、测试步骤、预期结果、优先级）
- 覆盖正常场景、异常场景、边界场景
- 编写完成后等待评审员反馈

**第3步 - 测试用例评审员（test_reviewer）**：
- 评审用例的完整性、准确性和可执行性
- 指出遗漏的测试场景
- 提出改进建议
- 判断：严重问题→标记 REVIEW_REJECT 要求重新编写；质量合格→标记 REVIEW_PASS 并 TERMINATE

**第4步 - 流程协调员（coordinator）**：
- 根据评审结果决定流程
- REVIEW_REJECT → 通知编写员重新编写
- REVIEW_PASS → 总结最终结果，发送 TERMINATE

### 3.3 消息过滤逻辑

- 后端在收集流式消息时，过滤掉 `source == "user_proxy"` 的中间消息
- 同一智能体的连续输出合并为一个 `AgentResponse`
- `TaskResult` 类型消息触发状态保存，但不推送给前端
- 前端 `MessageBubble` 同样过滤 `source === 'user_proxy'` 的消息

### 3.4 会话状态管理

```
创建会话 → sessions[session_id] = {
    queue: asyncio.Queue,       // 用户输入队列
    team: RoundRobinGroupChat,  // AutoGen 团队实例
    active: True,               // 激活状态
    waiting_for_input: False,   // 是否等待用户输入
    event_queue: Queue,         // SSE 事件队列（连接时创建）
    task: asyncio.Task          // 当前对话任务（运行时创建）
}
```

- 团队状态支持持久化到 `team_state.json`（JSON 格式）
- 对话历史持久化到 `team_history.json`
- 状态持久化为异步操作，不影响主流程

### 3.5 文件上传双端解析

**设计原因**：前后端各实现一次文件解析，前端解析用于即时展示附件信息，后端解析作为兜底确保 AI 能获取文件内容。

| 解析位置 | 技术实现 | 用途 |
|---------|---------|------|
| 前端 | pdfjs-dist（浏览器端） | 解析后暂存，随消息发送 |
| 后端 | PyMuPDF（服务端） | 通过 `/api/upload` 端点，可独立使用 |

## 四、非功能需求

### 4.1 性能要求

- SSE 流式响应：首字延迟 < 5 秒（取决于模型响应速度）
- 单次对话最大消息数：30 轮（由 MaxMessageTermination 控制）
- 前端消息列表无上限，支持滚动查看

### 4.2 可用性要求

- SSE 断开自动重连：最多 5 次，间隔 3 秒
- 后端无状态设计（session 在内存中），重启后需前端重新连接
- 前端 60 秒无事件时发送 keep-alive 保持连接

### 4.3 安全性

- 后端 CORS 配置允许所有来源（`allow_origins=["*"]`），生产环境需限制
- API Key 通过环境变量配置，不硬编码在代码中
- 文件上传仅支持 PDF/TXT，不做额外安全检查

## 五、边界场景与约束

| 场景 | 处理方式 |
|------|---------|
| 同一会话发送多条消息 | 拒绝：返回 "A conversation is already in progress" |
| SSE 流未建立时发消息 | 拒绝：返回 "No active SSE stream. Please refresh the page." |
| 无效 session_id | 返回 404 "Session not found" |
| 不支持的附件格式 | 返回错误 "仅支持PDF和TXT文件" |
| PDF 解析失败 | 抛出 ValueError，前端显示错误提示 |
| 模型 API 调用失败 | 错误信息通过 SSE 以 error 事件推送 |
| 配置缺失（无 API Key） | 启动时验证，抛出 ValueError |
