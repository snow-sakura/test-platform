# snow-platform 需求与业务逻辑

> 基于代码仓储全量分析

---

## 一、项目定位

**AI 测试用例生成平台** — 面向测试工程师的效率工具，核心目标是从 PRD（产品需求文档）中自动提取测试点并生成可执行的测试用例，减少人工编写测试用例的重复劳动。

---

## 二、用户需求分析

### 2.1 目标用户
- **测试工程师**：需要快速从 PRD 文档提取测试点、生成测试用例
- **测试经理**：需要管理多个项目的测试进度，确保测试覆盖
- **质量保障团队**：需要批量生成和维护测试资产

### 2.2 核心痛点
1. **手工编写测试用例效率低**：从 PRD 到测试用例需要大量人工分析
2. **测试覆盖不完整**：人工容易遗漏边界场景
3. **历史经验难复用**：过往 Bug、业务经验难以系统化融入新的测试
4. **多格式文档处理**：PRD 文档格式多样（PDF/DOCX/MD/YAML/CSV）
5. **团队协作通知**：测试任务完成后需要通知相关人员

### 2.3 需求全景

#### P0（核心链路，必须完成）
| 需求 | 说明 |
|------|------|
| 项目管理 | 创建/编辑/删除项目，项目列表展示统计信息 |
| 多格式文档上传 | 支持 PDF/DOCX/MD/YAML/CSV 上传，后台自动解析 |
| AI 提取测试点 | 基于文档内容，调用大模型自动提取测试点（含标题/描述/优先级/分类） |
| AI 生成测试用例 | 基于测试点，自动生成详细测试用例（含前置条件/步骤/预期结果） |
| 测试点/用例人工校验 | 手动编辑、确认、删除测试点和用例 |
| Excel 导出 | 将测试用例导出为标准格式的 Excel 文件 |

#### P1（重要功能）
| 需求 | 说明 |
|------|------|
| RAG 知识库 | 上传参考资料（历史 Bug、业务经验），增强 AI 生成质量 |
| 异步任务追踪 | 后台异步处理，前端实时轮询进度 |
| 飞书通知 | 任务完成后通过 Webhook 发送飞书消息 |
| 系统设置持久化 | API Key / 模型 / Webhook 配置持久化，运行时热更新 |

#### P2（增强功能）
| 需求 | 说明 |
|------|------|
| 自定义 LLM API 代理 | 支持硅基流动等第三方 LLM 平台 |
| PDF 乱码自动降级 OCR | macOS Quartz PDF 乱码时自动切换 OCR 识别 |
| 任务取消 | 支持取消正在运行的后台任务 |
| 多知识库关联 | 同时关联多个知识库增强生成 |

## 三、业务逻辑详解

### 3.1 端到端核心链路

```
[用户操作]                    [系统处理]
                                          
新建项目                     → 创建 Project 记录
  ↓                           
上传 PRD 文档                → 文档保存 + 格式自动解析 
（PDF/DOCX/MD/YAML/CSV）    → 解析文本存入 Document.content
  ↓
选择文档 + 可选关联知识库    → 创建 TaskBatch（PENDING）
  ↓                           
触发 AI 提取测试点           → BackgroundTask 异步处理：
  ├─ 更新批次为 RUNNING
  ├─ 遍历每个文档：
  │   ├─ 查询 RAG 上下文（若关联知识库）
  │   ├─ 调用大模型提取测试点（JSON 数组）
  │   └─ 写入 TestPoint 表 + 更新进度
  ├─ 更新批次为 COMPLETED
  └─ 发送飞书通知（可选）
  ↓
前端轮询进度                 → BatchTracker 每 3 秒查询
  ↓
人工校验测试点               → 查看/编辑/确认（is_verified=true）
  ↓
选择测试点 + 可选关联知识库  → 创建 TaskBatch
  ↓
触发 AI 生成测试用例         → 异步处理：
  ├─ 遍历每个测试点
  │   ├─ 查询 RAG 上下文
  │   ├─ 调用大模型生成用例（JSON 数组）
  │   └─ 写入 TestCase 表 + case_number 编号
  └─ 发送飞书通知
  ↓
查看/编辑测试用例            → 编辑、删除、批量删除
  ↓
导出 Excel                   → 生成标准格式 Excel 文件
```

### 3.2 业务规则

#### 项目规则
- 项目名称必填，长度 1~200 字符
- 删除项目时级联删除所有关联数据（文档、测试点、用例、批次）
- 项目列表返回统计数字（文档数、测试点数、用例数）

#### 文档规则
- 支持的文件类型：pdf、docx、md、markdown、yaml、yml、csv
- 上传时自动解析文档内容存入数据库，解析失败则删除已保存的物理文件
- 文件物理存储路径：`backend/uploads/`，UUID 前缀防重名
- 删除文档时同步删除物理文件

#### 测试点规则
- 每个测试点关联一个项目和一个可选来源文档
- 优先级：HIGH / MEDIUM / LOW（默认 MEDIUM）
- 分类：功能 / UI / 性能 / 安全 / 兼容性 等（自由填写）
- 支持人工校验确认（is_verified 字段）
- 删除测试点时级联删除关联的测试用例

#### 测试用例规则
- 用例编号格式：`TC-{test_point_id}-{seq}`
- steps 字段存储为 JSON 数组：`[{step, expected_result}]`
- 支持的功能：查看详情（含完整步骤）、编辑、删除、批量删除
- 导出 Excel 的列：用例编号 / 用例标题 / 前置条件 / 测试步骤 / 预期结果 / 优先级 / 用例类型

#### 知识库规则
- 创建知识库时自动在 ChromaDB 创建对应集合（UUID 命名）
- 上传文档到知识库时自动解析 → 分块（500 字符/块，50 字符重叠）→ 向量化 → 存入 ChromaDB
- 删除知识库时同步删除 ChromaDB 集合
- 知识库文档不支持单独删除（前端无操作入口，后端无独立 API）

#### 异步任务规则
- 使用 FastAPI BackgroundTasks（单进程，非 Celery）
- 后台任务必须使用独立数据库会话（AsyncSessionLocal）
- 每个文档/测试点单独 try-catch，单个失败不阻断整体
- 任务状态流转：PENDING → RUNNING → COMPLETED / FAILED
- 前端轮询间隔：3 秒（仅在有运行任务时轮询）
- 支持取消 PENDING/RUNNING 状态的任务

#### 系统设置规则
- 预定义配置键：LLM_API_KEY、LLM_MODEL、LLM_BASE_URL、FEISHU_WEBHOOK_URL
- 写入数据库后同步更新内存中的运行时配置（热更新）
- 应用冷启动时以 .env 文件初始值为准，不会自动从数据库恢复
- 前端启动时从 API 加载历史配置用于显示

#### 飞书通知规则
- 通知类型：测试点提取完成 / 测试用例生成完成
- 消息格式：飞书 post 富文本
- 触发时机：异步任务正常完成后自动触发
- 失败处理：仅记录日志，不阻断主流程

## 四、数据模型 ER 关系

```
┌─────────────┐       ┌────────────────┐
│   Project   │1──N──│   Document     │
│  (项目)     │       │  (项目文档)    │
└─────────────┘       └────────────────┘
      │1                     
      │
      ├──N──┌────────────────┐       ┌────────────────┐
      │     │   TestPoint    │1──N──│   TestCase     │
      │     │  (测试点)      │       │  (测试用例)    │
      │     └────────────────┘       └────────────────┘
      │
      └──N──┌────────────────┐
             │   TaskBatch    │
             │  (任务批次)    │
             └────────────────┘

┌────────────────┐       ┌──────────────────────┐
│ KnowledgeBase  │1──N──│ KnowledgeDocument    │
│ (RAG知识库)    │       │ (知识库文档)         │
└────────────────┘       └──────────────────────┘

┌──────────────────┐
│ SystemSettings   │  (独立表，按 key-value 存储)
│ (系统设置)       │
└──────────────────┘
```

**核心约束：**
- Project 与 Document/TestPoint/TestCase/TaskBatch 为 1:N 关系，级联删除
- TestPoint 与 TestCase 为 1:N 关系，级联删除
- TestPoint 可选关联 Document（document_id nullable）
- KnowledgeBase 与 KnowledgeDocument 为 1:N 关系，级联删除
- SystemSettings 按 key 唯一索引

## 五、用户交互流程

### 5.1 首次使用流程
```
1. 进入平台 → 项目列表页（空状态）
2. 点击"新建项目" → 填写名称和描述 → 创建成功
3. 点击项目卡片 → 进入项目详情
4. 默认进入「文档管理」Tab → 上传 PRD 文档
5. 选择文档 → 点击"AI 提取测试点" → 弹出知识库选择对话框（可选）
6. 确认后 → 跳转到「任务批次」Tab 查看进度
7. 任务完成后 → 自动弹窗通知
8. 切换到「测试点」Tab → 查看 AI 提取的测试点
9. 人工校验 → 编辑/确认测试点
10. 勾选测试点 → 点击"生成测试用例" → 可选关联知识库
11. 任务完成后 → 切换到「测试用例」Tab 查看
12. 可编辑/导出 Excel
```

### 5.2 使用 RAG 增强流程
```
1. 在项目内「知识库」Tab → 创建知识库
2. 上传参考资料（历史 Bug、业务经验文档等）
3. 在提取测试点或生成用例时关联该知识库
4. AI 生成时自动检索相关知识片段拼接到 Prompt
```

### 5.3 系统配置流程
```
1. 进入「系统设置」页面
2. 配置 LLM：API Key + 模型名称 + Base URL（可选）
3. 配置飞书：Webhook URL（可选）
4. 保存后即时生效（运行时热更新）
```

## 六、非功能性需求

| 维度 | 说明 |
|------|------|
| 性能 | 无分页，列表全量返回；异步任务基于单进程 BackgroundTasks |
| 安全 | 无用户认证/权限体系，单用户模式；API Key 明文存储 |
| 可用性 | 前端 Element Plus 中文界面；任务进度实时反馈 |
| 兼容性 | 支持多种 LLM 平台（OpenAI/Claude/DeepSeek/硅基流动等） |
| 可维护性 | 项目按领域拆分为 API 路由、Service 层、Models 层 |

---

## 七、已知功能缺口

1. 用户认证/权限系统 — 完全缺失
2. 数据分页 — 列表全部全量返回
3. 知识库文档单独删除 — 无独立 API
4. 测试用例再生成 — 无独立接口
5. 任务失败重试 — 无重试机制
6. 重新解析文档 — 无独立接口
7. WebSocket/SSE 替代轮询 — 任务进度依赖前端轮询
8. 应用冷启动时配置恢复 — 不会从数据库加载 SystemSettings
