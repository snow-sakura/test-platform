## 2. 核心模块设计

### 2.1 后端模块

#### `config.py` — 配置管理

**职责**：统一管理所有配置项，提供模型配置和验证功能。

```
Config 类（静态类）
├── DEEPSEEK_API_KEY    # DeepSeek API 密钥
├── DEEPSEEK_BASE_URL   # API 基础地址（默认：https://api.deepseek.com/v1）
├── DEEPSEEK_MODEL      # 模型名称（默认：deepseek-chat）
├── HOST               # 服务绑定地址（默认：0.0.0.0）
├── PORT               # 服务端口（默认：8000）
│
├── validate()          # 验证 API Key 是否配置
├── get_model_config()  # 返回模型连接配置字典
└── get_model_info()    # 返回模型能力信息（vision/function_calling等）
```

#### `sse_handler.py` — SSE 事件处理器

**职责**：管理会话生命周期、处理 SSE 流式推送、协调 AutoGen 对话流程。

```
SSEHandler 类
├── sessions: Dict[str, Dict]    # 会话池（内存管理）
│
├── create_session()             # 创建会话 + 初始化 AutoGen 团队
├── send_message()               # 接收用户消息，启动后台对话任务
├── _process_conversation()      # 核心：执行 AutoGen 流式对话
├── _send_to_session()           # 向 SSE 事件队列推送数据
├── stream_events()              # SSE 事件流生成器（FastAPI StreamingResponse）
└── _format_sse_event()          # 格式化 SSE 事件字符串

会话数据结构：
{
    "queue": asyncio.Queue,      # 用户输入队列（多轮对话用）
    "team": RoundRobinGroupChat, # AutoGen 团队实例
    "active": bool,              # 会话活跃状态
    "waiting_for_input": bool,   # 是否等待用户输入
    "task": asyncio.Task,        # 后台对话任务
    "event_queue": asyncio.Queue # SSE 事件队列
}
```

#### `team_manager.py` — AutoGen 团队管理器

**职责**：创建和管理 AutoGen 多代理团队，处理状态持久化。

```
TeamManager 类
├── team: RoundRobinGroupChat    # AutoGen 团队实例
├── state_file: str              # 团队状态文件（team_state.json）
├── history_file: str            # 聊天历史文件（team_history.json）
│
├── create_team(user_input_func) # 创建 3 个智能体 + RoundRobinGroupChat
├── _load_team_state()           # 从文件恢复团队状态
├── save_team_state()            # 保存团队状态到文件
├── save_history()               # 保存聊天历史
└── load_history()               # 加载聊天历史
```

**智能体配置**：

| 智能体 | system_message 职责摘要 |
|--------|------------------------|
| test_writer | 编写 Markdown 表格格式的测试用例，覆盖正常/异常/边界场景 |
| test_reviewer | 评审完整性、准确性和可执行性，标记 REVIEW_REJECT 或 REVIEW_PASS |
| coordinator | 管理流程，分配任务，总结结果，触发 TERMINATE |

**终止条件**：
```python
termination = TextMentionTermination("TERMINATE") | MaxMessageTermination(30)
```

#### `file_handler.py` — 文件处理

**职责**：解析上传的 PDF 和 TXT 文件，提取文本内容。

```
FileHandler 类（静态方法）
├── extract_text_from_pdf(content)  # 使用 PyMuPDF (fitz) 解析 PDF
└── extract_text_from_file(content, filename)  # 根据文件类型路由
```

#### `main.py` — API 路由

**职责**：FastAPI 应用入口，定义 REST API 端点。

| 端点 | 方法 | 功能 |
|------|------|------|
| `/health` | GET | 健康检查 |
| `/api/session` | POST | 创建新会话 |
| `/api/chat` | POST | 发送用户消息（支持 attachment 字段） |
| `/api/upload` | POST | 上传文件并提取文本 |
| `/api/stream/{session_id}` | GET | SSE 流式响应端点 |

### 2.2 前端模块

#### `useSSE.js` — SSE 连接管理（Composable）

**职责**：封装 SSE 连接、会话创建、消息发送、事件处理的全部逻辑。

```
useSSE()
├── connect()                     # 创建会话 + 建立 SSE 连接
├── disconnect()                  # 断开连接 + 清理资源
├── sendMessage({content, attachment})  # POST 发送消息
├── clearMessages()               # 清空消息列表
│
├── messages (ref)                # 消息列表（响应式）
├── isConnected (ref)             # 连接状态
├── isProcessing (ref)            # 处理中状态
├── isWaitingForInput (ref)       # 等待用户输入状态
└── sessionId (ref)               # 当前会话 ID
```

**SSE 事件处理流程**：

```
SSE Event: connected  → 确认连接成功
SSE Event: processing → 设置 isProcessing = true
SSE Event: AgentResponse → 解析智能体响应，添加到消息列表
SSE Event: done       → 设置 isProcessing = false
SSE Event: error      → 显示错误信息
```

**重连机制**：
1. 检测到连接关闭（readyState === CLOSED）
2. 重连次数 < 5 次 → 等待 3 秒后重新创建会话并连接
3. 超过 5 次 → 提示用户刷新页面

#### `fileParser.js` — 前端文件解析

**职责**：在浏览器端解析 PDF/TXT 文件。

```
parseFile(file)          # 入口：根据文件类型路由
├── parsePDF(file)       # 使用 pdfjs-dist 解析 PDF
└── parseTextFile(file)  # 使用 File API 读取文本文件
```

#### 组件结构

| 组件 | 职责 | 主要 Props |
|------|------|-----------|
| `ChatWindow.vue` | 主容器：管理消息列表、连接生命周期 | 无（内部使用 useSSE） |
| `MessageBubble.vue` | 单条消息渲染：时间轴、智能体标签、折叠 | `message: Object` |
| `InputArea.vue` | 输入框 + 文件上传 | `disabled, isProcessing, isWaitingForInput` |
| `AgentStatus.vue` | 顶部状态栏 | `isConnected, isProcessing` |

## 3. 数据流设计

### 3.1 消息发送与响应流程

```
[用户] → 输入消息 + (可选附件)
    │
    ▼
[InputArea.vue] → emit('send', {content, attachment})
    │
    ▼
[ChatWindow.vue] → sendMessage({content, attachment})
    │
    ▼
[useSSE.js] → POST /api/chat {session_id, content, attachment}
    │
    ▼
[main.py] → await sse_handler.send_message(session_id, content, attachment)
    │
    ▼
[sse_handler.py]:
    1. 检查 session 是否有效
    2. 如果等待用户输入 → 放入 input_queue
    3. 否则 → 创建 asyncio.create_task(_process_conversation)
    │
    ▼
[_process_conversation]:
    1. 有附件 → 拼接到用户消息前
    2. team.run_stream(task=user_message)
    3. 流式收集各智能体输出
    4. 按 source 分组合并
    5. 逐组推送 AgentResponse 事件
    6. 最终推送 done 事件
    │
    ▼
[SSE Stream] → 前端 EventSource 接收事件
    │
    ▼
[useSSE.js] → 解析事件 → messages.value.push(newMessage)
    │
    ▼
[MessageBubble.vue] → 渲染消息（带时间轴和智能体标识）
```

### 3.2 SSE 事件格式

```json
// 连接成功
event: connected
data: {"session_id": "xxx", "status": "connected"}

// 处理中
event: processing
data: {"status": "processing"}

// 智能体响应
event: AgentResponse
data: {
  "type": "AgentResponse",
  "source": "coordinator|test_writer|test_reviewer",
  "content": "智能体的回复内容",
  "timestamp": "2026-06-04T12:00:00"
}

// 对话完成
event: done
data: {"status": "completed"}

// 错误事件
event: error
data: {"type": "error", "content": "错误描述"}

// 等待用户输入（多轮对话）
event: UserInputRequest
data: {"type": "UserInputRequest", "content": "...", "source": "system"}
```

## 4. 关键设计决策

### 4.1 为什么选择 SSE 而不是 WebSocket？

- **单向推送**：AI 响应主要是服务端→客户端的单向数据流
- **浏览器原生支持**：EventSource API 原生支持自动重连
- **实现简单**：FastAPI 的 StreamingResponse 天然支持 SSE
- **HTTP 兼容性**：可穿透大部分代理和防火墙

### 4.2 为什么前端和后端都解析 PDF？

- **前端解析**：即时显示文件内容预览，减少网络传输
- **后端解析**：作为兜底，确保内容完整性（后端 `file_handler.py` + `/api/upload` 端点）

### 4.3 会话状态管理

- **运行时**：内存 Dict 存储（`sse_handler.sessions`）
- **持久化**：JSON 文件（`team_state.json` + `team_history.json`）
- **限制**：重启后会话丢失，需刷新页面重新创建

## 5. 安全性考虑

- **CORS**：当前配置为 `allow_origins=["*"]`，生产环境应限制具体域名
- **API Key**：通过 `.env` 文件配置，不硬编码在代码中
- **无认证**：当前版本无用户认证机制，适合内网或开发环境使用
