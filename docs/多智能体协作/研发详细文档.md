## 2. 后端核心代码详解

### 2.1 `main.py` — API 路由层

**文件路径**：`backend/app/main.py`

**设计模式**：模块化路由，将业务逻辑委托给 `SSEHandler` 和 `FileHandler`。

**关键代码解析**：

```python
# CORS 配置：当前允许所有来源，生产环境应限制
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

**注意**：生产部署时 `allow_origins` 应改为具体的前端域名列表。

```python
# ChatRequest 模型
class ChatRequest(BaseModel):
    session_id: str
    content: str
    attachment: Optional[str] = None  # 附件文本内容（已由前端解析）
```

**设计思路**：附件在前端已解析为文本，后端直接拼接使用，无需处理二进制文件。

### 2.2 `config.py` — 配置管理

**文件路径**：`backend/app/config.py`

**设计模式**：静态类 + `dotenv` 环境变量加载。

**关键说明**：

```python
@classmethod
def get_model_info(cls) -> dict:
    """模型能力信息（适配非 OpenAI 官方模型）"""
    return {
        "vision": False,           # DeepSeek 不支持图片识别
        "function_calling": True,  # 支持函数调用
        "json_output": True,       # 支持 JSON 输出
        "family": "unknown",       # 模型家族标识
        "structured_output": False,# 不支持结构化输出
    }
```

**为什么需要 `model_info`**：AutoGen 在创建 `OpenAIChatCompletionClient` 时需要知道模型的各项能力，尤其是 DeepSeek 等第三方模型与 OpenAI 官方模型存在差异时，需要明确告知 AutoGen。

### 2.3 `team_manager.py` — 多代理团队管理

**文件路径**：`backend/app/team_manager.py`

**设计模式**：单例实例（全局 `team_manager`）+ 工厂方法创建团队。

#### 三个智能体的分工与提示词设计

**test_writer（测试用例编写员）**：

```
职责：
1. 根据需求描述编写详细的测试用例
2. 测试用例应包含：用例编号、用例标题、前置条件、测试步骤、预期结果、优先级
3. 使用 Markdown 表格格式输出
4. 覆盖正常场景、异常场景和边界场景
```

**test_reviewer（测试用例评审员）**：

```
职责：
1. 评审测试用例的完整性、准确性和可执行性
2. 指出遗漏的测试场景
3. 提出改进建议
4. 问题严重 → REVIEW_REJECT + 要求重写
5. 质量合格 → REVIEW_PASS + TERMINATE
```

**coordinator（流程协调员）**：

```
职责：
1. 管理测试用例编写和评审流程
2. REVIEW_REJECT → 协调重写
3. REVIEW_PASS → 总结 + TERMINATE
4. 开始：接收需求并分配任务
```

#### 终止条件

```python
termination = TextMentionTermination("TERMINATE") | MaxMessageTermination(30)
```

- **正常终止**：任意智能体输出包含 "TERMINATE" 字符串
- **异常终止**：消息总数超过 30 条（防止无限循环）
- 两条件满足其一即终止

#### 状态持久化

```python
# 保存团队状态（AutoGen 内部状态）
await team.save_state() → team_state.json

# 保存聊天历史（人类可读）
await team_manager.save_history(messages) → team_history.json
```

### 2.4 `sse_handler.py` — SSE 事件处理（核心）

**文件路径**：`backend/app/sse_handler.py`

**设计模式**：事件驱动 + 异步队列（`asyncio.Queue`）。

#### 会话生命周期

```
create_session()
  ├── 生成 UUID session_id
  ├── 创建 asyncio.Queue（用户输入队列）
  ├── 创建 user_input_func（从队列获取输入）
  ├── team_manager.create_team(user_input_func)
  └── 存入 sessions 字典
      │
stream_events(session_id)
  ├── 创建 asyncio.Queue（事件队列）
  ├── yield connected 事件
  └── while True: 从事件队列取数据 → yield SSE 事件
      │
send_message(session_id, content, attachment)
  ├── 检查 session 有效
  ├── 如果 waiting_for_input → 放入 input_queue
  ├── 否则 → asyncio.create_task(_process_conversation)
      │
_process_conversation(session_id, content, attachment)
  ├── 有附件 → 拼接：附件内容\n\n用户需求：\n{content}
  ├── team.run_stream(task=user_message)
  ├── async for message in stream:
  │   ├── TaskResult → 保存状态 + 发送 done
  │   ├── user_proxy → 跳过
  │   └── 其他 → 按 source 分组收集
  └── 分组完成后 → 逐组发送 AgentResponse
```

#### 关键设计点

**为什么使用两个 Queue**：
- `input_queue`：处理多轮对话中等待用户输入的场景
- `event_queue`：向 SSE 流推送事件

**为什么对话在后台 Task 中运行**：
```python
task = asyncio.create_task(self._process_conversation(...))
```
防止对话处理阻塞 SSE 事件循环。

**为什么对话结束后不关闭 SSE 流**：
允许用户继续在同一会话中发送新消息，实现多轮对话。

### 2.5 `file_handler.py` — 文件处理

**依赖**：PyMuPDF（`fitz`）

**支持格式**：`.pdf`（使用 PyMuPDF 提取文本）、`.txt`（UTF-8 解码）

## 3. 前端核心代码详解

### 3.1 `useSSE.js` — SSE 连接管理

**文件路径**：`frontend/src/composables/useSSE.js`

**设计模式**：Vue 3 Composition API（组合式函数），封装 SSE 全部逻辑。

#### 连接流程

```
connect()
  ├── createSession() → POST /api/session → 获取 session_id
  └── new EventSource(url)
      ├── onopen → 标记已连接
      ├── addEventListener('connected') → 确认连接
      ├── addEventListener('processing') → 显示处理中
      ├── addEventListener('AgentResponse') → 添加到消息列表
      ├── addEventListener('done') → 完成处理
      ├── addEventListener('error') → 显示错误
      └── onerror → 自动重连（最多 5 次）
```

#### 重连机制

```javascript
// 关键代码
if (eventSource?.readyState === EventSource.CLOSED) {
    if (reconnectAttempts.value < maxReconnectAttempts) {
        reconnectAttempts.value++
        setTimeout(() => {
            sessionId.value = null  // 重置会话
            connect()               // 重新连接
        }, 3000)
    }
}
```

#### 消息发送

```javascript
const sendMessage = async ({ content, attachment }) => {
    // POST /api/chat { session_id, content, attachment }
    // 成功后添加用户消息到本地列表
    messages.value.push({
        id: Date.now(),
        content: content,
        source: 'user',
        timestamp: new Date().toISOString(),
        type: 'TextMessage'
    })
    isProcessing.value = true
}
```

## 4. 智能体协作流程详解

### 4.1 首次对话

```
用户: "帮我编写登录功能的测试用例"

1. [coordinator] 流程协调员：
   "收到需求，我来分配任务。
    @test_writer 请根据登录功能需求编写测试用例"

2. [test_writer] 测试用例编写员：
   "| 编号 | 用例标题 | 前置条件 | 测试步骤 | 预期结果 | 优先级 |
    |------|---------|---------|---------|---------|-------|
    | TC001 | 正常登录 | ... | ... | ... | P0 |"

3. [test_reviewer] 测试用例评审员：
   "评审意见：覆盖了主要场景，建议补充：
    - 密码连续错误锁定测试
    - 跨域登录场景
    REVIEW_REJECT"

4. [test_writer] 测试用例编写员：
   "已补充评审意见中提到的场景：
    | TC010 | 连续登录失败锁定 | ... |"

5. [test_reviewer] 测试用例评审员：
   "用例已覆盖主要场景，质量合格。
    REVIEW_PASS
    TERMINATE"

6. [coordinator] 流程协调员：
   "测试用例编写完成，以下是最终结果...
    TERMINATE"
```

### 4.2 含文件附件的对话

```
用户: "请根据附件内容编写测试用例"
附件: [需求文档.pdf 解析后的文本]

后端拼接后的消息：
"附件内容：
[需求文档的文本内容]

用户需求：
请根据附件内容编写测试用例"
```

### 4.3 多轮对话

```
第一轮：用户→"编写登录测试用例" → AI 完成
第二轮：用户→"再补充一个密码找回的用例" → AI 继续
```

## 6. 开发扩展指南

### 6.1 添加新的智能体

在 `team_manager.py` 的 `create_team` 方法中添加：

```python
# 创建新智能体
new_agent = AssistantAgent(
    name="new_agent_name",
    model_client=model_client,
    system_message="""自定义提示词""",
)

# 添加到参与者列表
team = RoundRobinGroupChat(
    participants=[coordinator, test_writer, test_reviewer, new_agent],
    termination_condition=termination,
)
```

在前端 `MessageBubble.vue` 中添加对应的显示配置：

```javascript
const agentConfig = {
  'new_agent_name': { label: '显示名', icon: '图标', color: '#颜色值' },
  // ...
}
```

### 6.2 更改 AI 模型

编辑 `backend/.env`：

```env
DEEPSEEK_API_KEY=新密钥
DEEPSEEK_BASE_URL=新API地址
DEEPSEEK_MODEL=新模型名
```

如需要调整模型能力信息，编辑 `config.py` 的 `get_model_info()`。

### 6.3 调整对话终止条件

在 `team_manager.py` 中修改：

```python
# 调整最大消息数
termination = TextMentionTermination("TERMINATE") | MaxMessageTermination(50)
# 或添加更多终止条件
# termination = termination | MaxMessageTermination(100)
```
